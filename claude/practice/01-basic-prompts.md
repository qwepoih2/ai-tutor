# 练习：基础提示词

## 目标
学会用黄金结构（角色 + 任务 + 约束 + 格式）提出问题，让 AI 输出可直接使用的内容。

## 约束
- 每个提示词都要包含：角色、任务、约束、输出格式
- 用中文写

## 资深开发场景题（选做）

以下 5 个场景来自日常开发，请选择 3 个，写出高质量提示词。

### 场景 A：Code Review
你写了一个函数，想让 AI 帮你 review。函数功能是：接收一个用户列表，过滤掉未激活用户，返回按注册时间排序的结果。

**用前端视角理解黄金结构：**

| 黄金结构 | 前端类比 | 作用 |
|---|---|---|
| **角色** | `props` — 告诉组件"你是谁" | 让 AI 切换到对应专家视角 |
| **任务** | 组件的核心逻辑 — "做什么" | 明确要完成的事 |
| **约束** | `validator` / 业务规则 — "怎么才算好" | 收窄输出范围，避免废话 |
| **格式** | UI 模板 — "长什么样" | 让输出可以直接用 |

**逐要素拆解：**

1. **角色**：`你是资深 JavaScript 工程师，擅长代码审查，关注性能、可读性和边界处理。` — 角色决定审查视角，就像团队里最注重性能的那个人 review 出来的意见和别人不同。
2. **任务**：贴上完整代码，而不是只描述功能。前端同学最容易犯的错——只描述不贴代码，AI 没有代码就只能空谈。
3. **约束**：指定重点检查项（时间复杂度、边界情况、类型安全），并要求"没问题就说没问题，不要硬凑建议"。约束越具体，输出越有用，就像你跟同事说"帮我看看性能"比"帮我看看"高效得多。
4. **格式**：表格 + 代码块。前端天天跟 UI 打交道，表格一目了然，直接能对照改代码。

**最终提示词：**

```
你是资深 JavaScript 工程师，擅长代码审查，关注性能、可读性和边界处理。

请对以下函数进行 Code Review：

function getActiveUsersSorted(users) {
  return users
    .filter(u => u.isActive)
    .sort((a, b) => new Date(a.createdAt) - new Date(b.createdAt));
}

要求：
1. 重点检查：时间复杂度、边界情况（空数组、undefined 字段）、类型安全
2. 如果发现问题，给出修改后的代码
3. 如果代码没问题，直接说"无明显问题"，不要硬凑建议
4. 不超过 300 字

输出格式：
- **问题列表**：表格形式，列：问题类型 / 严重程度(高/中/低) / 描述 / 修复建议
- **修改后代码**（如有）：完整可运行的函数
```

**对比：**

| | 原始问法 | 改写后 |
|---|---|---|
| **输入** | "帮我 review 这个函数" | 角色 + 代码 + 约束 + 格式 |
| **AI 的反应** | 泛泛而谈："注意边界" "命名可以更清晰" | 逐行分析，给出表格 + 修复代码 |
| **你能直接用吗** | 不能，还要追问 | 能，对照改就行 |
| **Token 浪费** | 高（来回追问 3-4 轮） | 低（一轮到位） |

### 场景 B：技术文档改写
你有一段内部 Wiki 文档（假设是你自己写的），需要改写成面向新人的 onboarding 文档。

**练习答案（初版）：**
> 你是一位资深的文档编写工程师，把这块wiki文档内容，改写成面向信任的文档，逻辑清晰、不啰嗦。

**批阅与问题诊断：**

| 问题 | 说明 |
|---|---|
| 错别字 | "面向信任" → "面向新人" |
| 缺少受众画像 | 新人的技术背景是什么？需要补哪些前置知识？ |
| 缺少结构指引 | onboarding 文档通常需要什么结构（快速开始、核心概念、FAQ）？ |
| 缺少改写规则 | 哪些保留、哪些删除、哪些需要补示例？ |
| "不啰嗦"太模糊 | 没有具体标准，模型不知道删到什么程度 |

**逐要素拆解：**

1. **角色**：`你是资深技术文档工程师，专注于新人 onboarding 体验。` — 角色决定改写视角，就像团队里最懂新人痛点的那个人。
2. **受众**：`刚入职的初级开发者，熟悉基础编程但不了解本项目技术栈和业务背景。` — 明确新人水平，模型才知道哪些术语要解释、哪些不用。
3. **任务**：贴上 Wiki 原文，明确要改写成 onboarding 文档。不要只描述"帮我改写"，要贴上原文让模型有内容可改。
4. **约束**：指定结构模板（快速上手 → 核心概念 → FAQ）、术语处理规则、精简标准，就像写组件时定义 props 类型一样具体。
5. **格式**：Markdown + 层级标题，直接可用。

**最终提示词：**

```
你是资深技术文档工程师，专注于新人 onboarding 体验。

任务：将以下内部 Wiki 改写成面向新人的 onboarding 文档。

受众：刚入职的初级开发者，熟悉基础编程但不了解本项目的技术栈和业务背景。

改写要求：
1. 结构：开头给出"5 分钟快速上手"，然后是核心概念，最后是常见问题
2. 术语：首次出现的专业术语必须加括号解释
3. 示例：每个操作步骤配一个最小可运行的代码/命令示例
4. 精简：删除项目历史、设计决策讨论等对新人无用的内容
5. 语气：用"你"直接对话，避免被动语态

输出格式：Markdown，使用清晰的层级标题。

[在此粘贴 Wiki 文档]
```

**对比：**

| | 原始问法 | 改写后 |
|---|---|---|
| **输入** | "把 wiki 改写成面向新人的文档，不啰嗦" | 角色 + 受众 + 结构 + 约束 + 格式 |
| **AI 的反应** | 随意精简，可能删掉关键内容，结构混乱 | 按模板输出，术语有解释，步骤有示例 |
| **你能直接用吗** | 不能，还要手动调整结构和补充内容 | 能，直接粘贴到 onboarding 仓库 |
| **Token 浪费** | 高（来回追问 2-3 轮补结构、补示例） | 低（一轮到位） |
